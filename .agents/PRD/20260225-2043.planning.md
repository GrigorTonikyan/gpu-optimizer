## 1. Project Overview

**Name:** Universal GPU Optimizer (TUI Improvements & Architecture Decoupling)
**Environment:** Linux (Distribution Agnostic)
**Runtime:** Bun + TypeScript
**Goal:** Address several critical UI/UX shortcomings identified in `docs/notes-todos.md`. The focus is on ensuring a robust, responsive, and fully decoupled TUI that is non-blocking, handles missing system dependencies gracefully, and provides an excellent user experience through path auto-completion and configurable defaults.

### What This PRD Covers

| Requirement | Description |
|---|---|
| Dependency Detection | App detects missing OS dependencies (such as fonts for icons displayed in terminal) and interactively prompts the user for permission to install them. |
| Filesystem Module | A centralized filesystem service/module for handling path traversals, auto-completion of paths in the TUI, and managing application data directories. |
| UI/Backend Decoupling | Complete decoupling of the TUI from the backend data loading processes. TUI navigation is immediate (non-blocking). |
| Background Pollers | Implementation of background data loading using Bun `Worker` (Web Workers api equivalent) or a robust async loop so the event loop remains unblocked and TUI UI remains responsive during data polling. |
| Configurable Paths | Users can specify custom, absolute paths for app data, config, logs, and backups, overriding default XDG paths. Includes support for multiple backup paths (with a primary default). |
| Standardized Logging | Replacing the numeric verbosity levels with standardized log levels: `fatal`, `error`, `warn`, `info`, `debug`, `trace`. Default dev is `debug`, prod is `info`. |
| Dynamic Resize Handling | The TUI (using `terminal-kit`) will properly detect terminal resize events, dynamically repaint the layout (handling data cut-off), and support scrolling. |

## 3. Core Architecture

### 3.1 Fully Decoupled UI and Backend

The communication between the TUI and Backend must be fully asynchronous. Navigation actions in the TUI trigger state changes instantly, showing a loading indicator (or skeleton UI) if data is missing, while dispatching requests to a background data provider.

```
┌─────────────────┐       events/requests      ┌──────────────────────┐
│   TUI Rendering │ ─────────────────────────► │ Background Service   │
│ (terminal-kit)  │ ◄───────────────────────── │ (Bun Worker / Async) │
└────────┬────────┘       data updates         └────────┬─────────────┘
         │ (repaints on data change & resize)           │
         ▼                                              ▼
   Terminal Output                             Filesystem / Sysfs / OS
```

### 3.2 Centralized Filesystem Service

A dedicated module (e.g., `src/utils/fs.ts` or `src/services/fs.ts`) must handle all file I/O operations, ensuring consistent path resolution, safety checks, and autocompletion traversal logic.

## 4. Configuration & Path Management

### 4.1 Custom Configuration Schema

The `AppConfig` interface will be updated to reflect the new requirements. Paths can be relative, absolute, or omitted (yielding to XDG defaults).

```typescript
type LogLevel = 'fatal' | 'error' | 'warn' | 'info' | 'debug' | 'trace';

interface AppConfig {
  /** Verbosity level: default debug (dev), info (prod) */
  verbosity: LogLevel;
  /** Custom path overrides (absolute paths allowed) */
  paths: {
    config?: string; // default: XDG_CONFIG_HOME/gpu-optimizer/config.json
    data?: string;   // default: XDG_DATA_HOME/gpu-optimizer/
    logs?: string;   // default: XDG_STATE_HOME/gpu-optimizer/logs/
  };
  backupPaths: {
    primary: string; // The directory where new backups are saved
    sources: string[]; // Additional directories to scan for available backups
  };
}
```

## 5. File Path Autocompletion

The centralized filesystem service must actively traverse user directories when the user types in path input fields within the TUI (such as adding a new backup directory or custom config path).

- The autocompletion must be instantaneous and non-blocking.
- Only directories should be suggested if the context is selecting a directory.
- `terminal-kit` `InputField` with custom autocomplete logic will be utilized.

---

## 6. Dependency Installation Manager

On startup (or upon first running the TUI), the app checks for required system dependencies such as Nerd Fonts (for kitty/alacritty etc. icon support).

- **Detection Phase:** Check if the host OS has necessary fonts installed. (e.g., `fc-list | grep -i "nerd\|symbols"`).
- **Prompt Phase:** Use TUI/CLI to display an interactive prompt. Tell the user what is missing and ask for permission to install it.
- **Execution Phase:** Run OS-specific package manager commands (or fallback to user-level `.local/share/fonts` download mechanism) seamlessly.

## 7. Implementation Stages

### Stage 13: Centralized FS and Paths
- [ ] Create `src/services/fs.ts` standardizing absolute/XDG path resolutions.
- [ ] Implement active filesystem traversal autocompletion logic in `fs.ts`.
- [ ] Update config schema and loader to support custom paths and primary/sources backup arrays.
- [ ] Refactor existing file operations to route through the new FS service.

### Stage 14: Standardized Logging System
- [ ] Replace numeric verbosity `(0|1|2)` with `LogLevel` (`fatal|error|warn|info|debug|trace`).
- [ ] Implement a unified logger (with or without external deps) that respects the configuration levels and outputs to the configured log path.

### Stage 15: Dependency Detection System
- [ ] Create `src/discovery/dependencies.ts` which exposes `checkSystemDependencies()`.
- [ ] Add rules for font detection (e.g., `fc-list` probing for Nerd Fonts).
- [ ] Add an interactive installation prompt/workflow leveraging `utils/shell.ts`.

### Stage 16: Decoupled Data Workers
- [ ] Extract hardware telemetry tasks (CPU, GPU polling) into a Bun Worker script (`src/workers/telemetry.worker.ts`).
- [ ] Establish message-passing protocol between the main thread (Controller Layer) and the Worker.
- [ ] Update Controllers to return immediate Promises or Subscriptions via the worker instead of synchronously blocking.

### Stage 17: TUI Responsiveness and Resize Handling
- [ ] Refactor `terminal-kit` screen implementations to subscribe to Worker updates. Screens should render skeletons/loaders while awaiting data.
- [ ] Add listener for `terminal.on('resize', ...)` to trigger full logical layout recalculations and clear/repaint.
- [ ] Ensure `terminal-kit` document layouts use flexible metrics (percents, flex options) rather than hardcoded dimensions, preventing truncation.
- [ ] Add path auto-completion fields to TUI configuration menus utilizing the new `fs.ts` traversal logic.

## 9. Success Criteria

1. **Dependency Installation:** App correctly identifies missing fonts and can download/install them if the user approves.
2. **FS Autocompletion:** Path input fields actively navigate real system directories without blocking the UI.
3. **Decoupled Backend:** TUI renders main layout in `< 50ms` on startup, with actual CPU/GPU metrics loading cleanly via Bun Worker soon after.
4. **Resizing:** Terminal resize does not break the layout or hide data irreversibly. Dynamic flexible box models work.
5. **Config Management:** App creates backups strictly in `backupPaths.primary`, but lists/restores backups successfully from paths in `backupPaths.sources`.
6. **Logging:** Setting `verbosity` to `trace` logs verbose output to the properly overridden log path, maintaining XDG defaults if unchanged.
